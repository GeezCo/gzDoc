## 🔄 服务间调用关系

### 场景1: 金融研报上传处理

```
用户上传研报
    ↓
API Gateway (8080)
    ↓
gzdoc-finance (8084) ← 场景层入口
    ↓
    ├─→ gzdoc-document (8082) ← 复用：文件上传
    │   └─→ MinIO ← 存储文件
    │
    ├─→ Kafka ← 发送消息: finance-report-upload
    │
    └─→ AI Service (8000)
        ├─→ 研报解析 (finance/report_parser.py)
        ├─→ 表格提取 (finance/table_extractor.py)
        ├─→ 实体识别 (finance/entity_recognizer.py)
        ├─→ Embedding (通用能力)
        └─→ Weaviate (FinanceReport Schema)
    ↓
保存到 PostgreSQL
    ├─→ t_document (平台表)
    └─→ t_finance_report (场景表)
    ↓
返回结果
```

### 场景2: 金融问答

```
用户提问: "茅台的直营占比是多少？"
    ↓
API Gateway (8080)
    ↓
gzdoc-finance (8084) ← 场景层入口
    ↓
    ├─→ 场景过滤: 只查询茅台的研报
    │   SELECT id FROM t_finance_report WHERE stock_code='600519'
    │
    └─→ gzdoc-qa (8083) ← 复用：问答能力
        └─→ AI Service (8000)
            ├─→ 向量检索 (Weaviate: FinanceReport)
            ├─→ 全文检索 (ES)
            ├─→ Rerank
            └─→ LLM生成答案
    ↓
场景增强 (gzdoc-finance)
    ├─→ 添加最新财务数据
    ├─→ 添加估值数据
    └─→ 添加历史趋势
    ↓
返回增强后的答案
```

## 💡 核心设计原则

### 1. 依赖方向

```
场景层 ──依赖──> 平台层
   ↓              ↓
 不能反向依赖   不感知场景层
```

**示例：**
```java
// ✅ 正确：场景层依赖平台层
@Service
public class FinanceReportService {
    @Autowired
    private DocumentService documentService;  // 平台层服务
}

// ❌ 错误：平台层不能依赖场景层
@Service
public class DocumentService {
    @Autowired
    private FinanceReportService financeService;  // 禁止！
}
```

### 2. 数据关联

```
平台表 (t_document)
    ↑ 外键关联
场景表 (t_finance_report)
```

**示例：**
```sql
-- 场景表通过 document_id 关联平台表
CREATE TABLE t_finance_report (
    id BIGSERIAL PRIMARY KEY,
    document_id BIGINT NOT NULL,
    stock_code VARCHAR(20),
    ...
    FOREIGN KEY (document_id) REFERENCES t_document(id)
);
```

### 3. 能力复用

```
场景层通过依赖注入复用平台层能力
不复制代码，不修改平台层
```

**示例：**
```java
// 场景层复用平台层能力
@Service
public class FinanceReportService {
    @Autowired
    private DocumentService documentService;    // 文档管理
    @Autowired
    private MinioService minioService;          // 对象存储
    @Autowired
    private EmbeddingService embeddingService;  // 向量化
    @Autowired
    private QAService qaService;                // 问答能力
    
    // 场景专用逻辑
    public void processReport(Long reportId) {
        // 复用 + 扩展
    }
}
```

## 🎯 优势总结

### vs 单体架构
- ✅ 模块化：场景独立，互不影响
- ✅ 可扩展：新增场景不改老代码
- ✅ 可维护：职责清晰，易于理解

### vs 传统微服务
- ✅ 更灵活：按场景组合部署
- ✅ 更经济：不用的场景不部署
- ✅ 更快速：场景开发并行进行

### vs 纯插件架构
- ✅ 更稳定：平台层提供稳定基础
- ✅ 更高效：复用通用能力
- ✅ 更简单：不需要复杂的插件机制
